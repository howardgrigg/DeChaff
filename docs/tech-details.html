Metadata extraction

Three-tier YouTube title parsing

When a video is selected for download, DeChaff immediately tries to extract structured sermon metadata from the YouTube video title — sermon title, Bible reading, preacher, series, and date. It attempts three strategies in order, falling back to the next if the previous one is unavailable or fails. The extraction runs concurrently with the audio download, so the fields are pre-filled by the time the download finishes.

1
Apple Intelligence On-device · FoundationModels framework · macOS 26+
Preferred

The first attempt uses Apple's FoundationModels framework to run a small language model entirely on-device. DeChaff creates a LanguageModelSession with a system prompt that describes the expected title format (injected from the user's Settings → Templates configuration), then asks the model to populate a structured SermonMetadata schema using the @Generable / @Guide macros. The model returns a strongly-typed Swift struct directly — no JSON parsing required.

Because it runs on the Neural Engine, it adds no network latency and costs nothing per request. It is skipped if Apple Intelligence is unavailable on the device (checked via SystemLanguageModel.default.availability). 

// Structured output schema — the model fills this in directly
@Generable
struct SermonMetadata {
    @Guide(description: "The sermon title or topic.")
    var title: String
    @Guide(description: "Bible reading, e.g. 'Romans 8:28'.")
    var bibleReading: String
    @Guide(description: "Preacher or speaker name.")
    var speaker: String
    @Guide(description: "Sermon series name.")
    var series: String
    @Guide(description: "Date as 'dd MMM yyyy'.")
    var date: String
}

let session = LanguageModelSession(
    model: SystemLanguageModel.default,
    instructions: systemInstructions(titleFormat: titleFormat)
)
let response = try await session.respond(
    to: "Extract sermon metadata from: \"\(title)\"",
    generating: SermonMetadata.self
)
2
Claude API Network · Anthropic API · requires API key in Settings
Fallback

If Apple Intelligence is unavailable, DeChaff falls back to the Anthropic Claude API — the same API used by the AI Assistant feature. It sends the same system prompt (with the user's title format injected) and asks for a JSON response with the five metadata fields. The response is parsed from JSON into a SermonMetadata struct.

This tier requires an API key to be configured in Settings → AI Assistant and an active internet connection. It is skipped silently if no key is present. The model used is whatever is selected in Settings (defaulting to Claude Sonnet 4.6). 

// Same system prompt as tier 1, with a JSON output instruction appended
let response = try await ClaudeAPIClient.sendMessage(
    apiKey: apiKey,
    model: model,
    systemPrompt: systemInstructions(titleFormat: titleFormat) + """

        Respond with ONLY a JSON object:
        {"title":"...","bibleReading":"...","speaker":"...","series":"...","date":"..."}
        """,
    transcript: "Extract sermon metadata from: \"\(title)\""
)
3
Template parser On-device · pure string matching · no AI required
Last resort

If both AI tiers fail or are unavailable, DeChaff falls back to a deterministic regex parser that uses the user's title format template directly. Unlike AI, it cannot handle variation or imprecision — but when titles are consistently formatted it works reliably with zero latency and zero cost. See Tier 3 detail below.

The shared system prompt

Tiers 1 and 2 use identical instructions. The user's title format is injected at runtime so the model understands the specific structure your church uses. This is why configuring the format in Settings → Templates matters — it significantly improves extraction accuracy for both AI tiers. 

"""
You extract structured metadata from church sermon YouTube video titles.

The expected format for this church is: "\(titleFormat)"
where {title} is the sermon title or topic, {reading} is the Bible reading or passage,
{preacher} is the speaker or preacher name, {series} is the sermon series name,
and {date} is the date of the sermon.

Fields may be separated by pipe characters (|), commas, dashes, or other delimiters
as shown in the format above. Not all fields are always present.

The date may appear in various formats (e.g. "23 Mar 2025", "15/3/26").
Always normalise it to "dd MMM yyyy" format.

Only populate fields you are confident about. Use empty string for anything not clearly
present. Never invent information.
"""

Tier 1 — Apple Intelligence

DeChaff uses the FoundationModels framework introduced in macOS 26 to access Apple's on-device language model via SystemLanguageModel.default. Before attempting extraction, it checks SystemLanguageModel.default.availability — if the result is anything other than .available (e.g. the device lacks Apple Intelligence, or the user hasn't enabled it), this tier is skipped immediately and Tier 2 is tried instead.

Rather than asking the model to produce free-form text and then parsing it, DeChaff uses the @Generable macro to declare SermonMetadata as a structured output schema. Each field is annotated with @Guide. The session.respond(to:generating:) call constrains the model's output to exactly this shape — DeChaff receives a fully-typed Swift struct with no JSON parsing step. Because the model runs on the Neural Engine, extraction adds no network latency and the YouTube title never leaves the device. 

let session = LanguageModelSession(
    model: SystemLanguageModel.default,
    instructions: systemInstructions(titleFormat: titleFormat)
)
let response = try await session.respond(
    to: "Extract sermon metadata from this YouTube title: \"\(title)\"",
    generating: SermonMetadata.self
)
return response.content  // already a SermonMetadata — no JSON parsing

Tier 2 — Claude API

DeChaff looks up the Anthropic API key from the macOS Keychain. If no key is stored, this tier is skipped immediately. Otherwise, it sends the same system prompt as Tier 1 with one addition: an instruction to respond with a plain JSON object containing the five metadata fields. The model used is whatever is selected in Settings → AI Assistant (defaulting to Claude Sonnet 4.6).

The response is trimmed and parsed with JSONSerialization into a [String: String] dictionary. Any key not present in the response becomes an empty string. If the JSON parse fails for any reason the tier returns nil and Tier 3 is attempted. Cloud models handle unusual date formats and ambiguous titles better than the on-device model — if your church's titles are consistently tricky, configuring an API key is worth it. 

let response = try await ClaudeAPIClient.sendMessage(
    apiKey: apiKey, model: model,
    systemPrompt: systemPrompt,  // same as tier 1 + JSON output instruction
    transcript: "Extract sermon metadata from this YouTube title: \"\(title)\""
)
guard let data = response.trimmingCharacters(in: .whitespaces).data(using: .utf8),
      let json = try JSONSerialization.jsonObject(with: data) as? [String: String]
else { return nil }

Tier 3 — Template parser

The last-resort path: a deterministic regex parser that uses the user's format template directly. No AI, no network, no latency — and no flexibility. When titles are consistently formatted it works perfectly; when they're not, the fields it can't match are left blank for the user to fill in.

Walking through the default template as a concrete example:

Input template
{title}, {reading} | {preacher} | {series} | {date}
1

Find the placeholders

The regex \{(\w+)\} scans the format string and returns each {…} match in document order, capturing the field name inside the braces. 

match 0  {title}     name="title"
match 1  {reading}   name="reading"
match 2  {preacher}  name="preacher"
match 3  {series}    name="series"
match 4  {date}      name="date"
2

Build segments

For each placeholder match, the code slices out the literal text that sits between the previous match's end and this match's start. That literal is whatever separator the user typed — it could be , , | , - , — , or anything else. 

segment 0  literal=""     field="title"    ← nothing before {title}
segment 1  literal=", "   field="reading"  ← text between {title} and {reading}
segment 2  literal=" | "  field="preacher" ← text between {reading} and {preacher}
segment 3  literal=" | "  field="series"
segment 4  literal=" | "  field="date"
3

Build the regex

Each segment contributes two things to the pattern: its literal text regex-escaped (so | becomes \|, . becomes \., etc.), then a capture group. All capture groups are non-greedy (.+?) except the last, which is greedy (.+) so it consumes everything to the end of the string. 

segment 0  ""     →            (.+?)
segment 1  ", "   → ,\       + (.+?)
segment 2  " | "  → \ \|\    + (.+?)
segment 3  " | "  → \ \|\    + (.+?)
segment 4  " | "  → \ \|\    + (.+)   ← greedy on the last field

→ final pattern:
^(.+?), (.+?) \| (.+?) \| (.+?) \| (.+)$

Non-greedy groups are critical here — each (.+?) matches as little as possible, stopping at the first occurrence of the next literal rather than consuming it.

4

Progressive fallback for missing fields

If the title has fewer fields than the template — for example, no date — the full regex won't match because the expected separator and capture group aren't there. The parser retries with one fewer trailing segment, rebuilding the regex each time, until it either finds a match or runs out of options. A minimum of two matching fields is required to prevent the degenerate case of ^(.+)$ matching any arbitrary string. 

-- title: "Grace Abounding, Romans 5:1-5 | Pastor Mike | Core Series"
   (has series but no date)

count=5  ^(.+?), (.+?) \| (.+?) \| (.+?) \| (.+)$  → no match
count=4  ^(.+?), (.+?) \| (.+?) \| (.+)$           → match ✓

title    = "Grace Abounding"
reading  = "Romans 5:1-5"
preacher = "Pastor Mike"
series   = "Core Series"   ← captured by the greedy group
date     = ""               ← not in title, defaults to empty
5

Map captures to fields

Each capture group maps back to a field name via the used segments array, which preserves the original order. Each value is whitespace-trimmed, then assembled into a SermonMetadata struct.

Result — default template, all fields present
title→"The Good Shepherd"
reading→"John 10:1–18"
preacher→"Rev. James Hart"
series→"Foundations"
date→"6 Apr 2026"

Template parser — known limitations

Because the parser is purely structural, it has two inherent limitations that AI tiers do not:

Separator collision. If the separator character appears inside a field value — for example a sermon titled "Faith, Hope, and Love" when the separator between {title} and {reading} is , — the non-greedy capture will stop at the first comma and the mapping will be wrong. Tiers 1 and 2 handle this correctly because they understand context rather than pattern-matching literally.

Position-dependence. Fields must appear in exactly the order the template specifies. A missing field in the middle of the template (rather than at the end) will cause all subsequent fields to shift and map incorrectly. Missing trailing fields are handled gracefully by the progressive fallback.

Audio processing

The processing pipeline

When you hit Process, DeChaff runs the trimmed audio through up to six sequential passes. Each pass writes a temporary WAV file that feeds the next, so no pass has to hold the entire recording in memory at once. Temporary files are deleted as soon as they are no longer needed. 

Input audio (any format)
    │
    ├─ Pass 1 ──── Voice isolation + dynamic compression  (AU render loop)
    │
    ├─ Pass 1.5 ── Slow leveler                           (optional)
    │
    ├─ Pass 2 ──── Loudness measurement (EBU R128)        │
    │              + gain application + soft limiter       ┘ (optional, combined)
    │
    ├─ Pass 2.5 ── Silence shortening                     (optional)
    │
    └─ Pass 3 ──── MP3 encoding via LAME                  (optional — WAV output skips this)
         │
         └── Output .mp3 (or .wav)
Pass 1a
Voice isolation AUSoundIsolation · subtype 0x766f6973

Apple's AUSoundIsolation Audio Unit runs the audio through an on-device neural model that separates speech from everything else — room reverb, chair scrapes, HVAC hum, crowd sounds, and anything that isn't a human voice. For a church recording, this is the single biggest quality improvement: a live room can add 300–500 ms of reverb tail to every word, and a distant lapel mic picks up the whole room equally.

The AU is pulled by subtype 0x766f6973 ('vois' in four-character code notation). Three parameters are set before initialisation:

ParameterValueWhy
param 0100.0 Isolation strength — 100 is full isolation. Lower values blend the cleaned signal with the original; 100 gives the most aggressive noise removal, which is correct for speech-only podcast use.
param 957821.0 Undocumented Apple parameter — empirically required to enable the neural model. Without it the AU passes audio through unmodified.
param 957831.0 Undocumented Apple parameter — similarly required. Both flags are present in Apple's own FaceTime and Voice Memo implementations.

Processing is chunk-based: 4096 frames (~85 ms at 48 kHz) are read, pushed through the AU's render callback, and written to a temporary WAV file. This keeps memory use flat regardless of recording length.

Pass 1b
Dynamic compression kAudioUnitSubType_DynamicsProcessor

The compressor runs in the same AU render loop immediately after voice isolation, on the cleaned signal. Church recordings have a high crest factor — the ratio between peak level and average level — because preachers vary their delivery significantly: quiet reflective passages, sudden emphatic words, prayer, congregational response. Compression reduces this range so that the loudness normalisation pass has less gap to close and the result sounds more consistent throughout.

ParameterValueWhy
Threshold−28 dBFS Set low enough to engage on all speech, including a quiet scripture reader or someone stepping back from the microphone. A higher threshold (e.g. −18 dB) would miss quieter passages entirely.
HeadRoom6 dB The "knee" above the threshold before full compression kicks in. A 6 dB headroom against a −28 dB threshold produces an effective ratio of approximately 4.7:1 — standard for podcast voice processing. Gentler than broadcast limiting (10:1+) but harder than music (2:1).
OverallGain+8 dB Makeup gain applied after compression. Because compression reduces the average level, makeup gain restores it. +8 dB brings the compressed speech closer to the loudness target, reducing the gain the normalisation pass has to apply — which in turn reduces how hard the soft limiter has to work.
AttackTime3 ms Fast enough to catch sharp transients (a hand clap near the mic, a cough) before they clip the headroom ceiling, but not so fast that it bites into consonants and makes speech sound dull. At 3 ms, /p/ and /t/ transients are largely preserved.
ReleaseTime150 ms Snappy enough to recover gain between words without audible pumping on continuous speech. 150 ms is roughly the gap between spoken syllables, so the compressor breathes naturally with the delivery.
ExpansionRatio1.0 Expansion ratio of 1.0 disables downward expansion (noise gating). The AU is used purely as a compressor; any noise gating would conflict with the voice isolation pass that already cleaned the signal.
Pass 1.5
Slow leveler optional · 1-second window RMS · ±6 dB · zero-phase EMA

Dynamic compression handles short-term peaks (syllable to syllable), but it can't fix a sustained level difference that lasts minutes — for example, a scripture reader who is consistently 10 dB quieter than the preacher throughout their entire reading. The slow leveler addresses this by building a smooth gain envelope across the whole recording and applying it in a second pass.

The algorithm runs in five stages:

1

Measure RMS per 1-second window

The file is scanned in 1-second blocks. The RMS across all channels is computed for each block using vDSP_measqv.

2

Gate and find the reference level

Windows below −40 dBFS are excluded — after voice isolation, residual room noise and breath noise sit around −40 to −45 dBFS. The median RMS of the remaining voiced windows becomes the reference level. Median (rather than mean) is used so that brief loud events like congregational laughter don't skew the reference upward.

3

Compute per-window gain, clamped to ±6 dB

Each voiced window gets a gain of reference / rms, clamped to ±6 dB. The ±6 dB limit prevents the leveler from over-correcting: a speaker who is genuinely quiet shouldn't be boosted into sounding unnaturally loud. Silent windows inherit the gain of the nearest voiced neighbour rather than snapping back to unity, which prevents the envelope from dipping through quiet gaps.

4

Smooth with zero-phase EMA (α = 0.25, ~3 s time constant)

A forward EMA pass (each window = 0.25 × its gain + 0.75 × the previous window's smoothed gain) is followed by a backward EMA pass. Running the EMA in both directions produces a zero-phase filter — the smoothed envelope is centred on the transitions rather than lagging behind them, which means the gain starts adjusting before a level change rather than reacting to it. The 0.25 coefficient corresponds to a time constant of roughly 3 seconds at 1-second window resolution.

5

Apply with per-window linear ramps

The gain is applied sample-by-sample using a linear ramp that interpolates from the midpoint between the previous window's gain and the current window's gain, to the midpoint between the current and the next window's gain. This guarantees continuity at window boundaries — no clicks or discontinuities.

Pass 2a
Loudness measurement optional · ITU-R BS.1770-4 / EBU R128

Before gain can be applied, the integrated loudness of the file must be measured in LUFS (Loudness Units relative to Full Scale). DeChaff implements the EBU R128 algorithm from scratch rather than relying on an external tool, so the measurement is accurate and runs entirely in-process.

The EBU R128 measurement has three stages:

K-weighting filter

Each channel is passed through a two-stage biquad cascade that approximates the frequency sensitivity of the human ear. Stage 1 is a high-shelf pre-filter (+4 dB above ~1.5 kHz — ears are more sensitive to high frequencies). Stage 2 is a high-pass RLB filter (rolls off below ~38 Hz — bass content is perceptually less loud). The coefficients are computed analytically for the input file's sample rate, so the filter is accurate at 44.1 kHz, 48 kHz, and any other rate. Processing uses Apple's vDSP_biquad for performance; only ~93 KB of K-weighted samples are held in memory at any time regardless of recording length.

Overlapping 400 ms blocks

The K-weighted signal is divided into 400 ms blocks at 100 ms hops (75% overlap). The mean square of each block is converted to a momentary LUFS value. Overlapping blocks increase temporal resolution without losing the 400 ms integration window required by the standard.

Two-pass gating

Absolute gate (−70 LUFS): blocks below −70 LUFS are discarded. This excludes complete silence — gaps between songs, dead air at the start and end of the recording.

Relative gate (ungated mean − 10 LU): a second pass discards blocks more than 10 LU below the ungated mean. This excludes quiet background noise that passed the absolute gate but shouldn't contribute to the measurement of speech loudness. The result is an integrated LUFS figure that accurately represents the perceived loudness of the speech content only.

Pass 2b
Gain application + soft limiter optional · −16 LUFS target · −1 dBFS ceiling

A single gain value (target LUFS − measured LUFS, in dB) is applied to bring the integrated loudness to the target. The default target is −16 LUFS, which is the EBU R128 recommendation for podcast and streaming platforms. A −1 dBFS true peak ceiling is enforced to leave headroom for MP3 encoder inter-sample peaks.

Rather than hard-clipping peaks that exceed the ceiling, a per-sample gain envelope follower (a soft limiter) is used:

ParameterValueWhy
Peak ceiling−1 dBFS MP3 encoding can introduce inter-sample peaks up to +3 dBFS above the digital peak due to the reconstruction filter. A −1 dBFS ceiling ensures the decoded MP3 stays below 0 dBFS on any playback device.
Attack1 ms Fast enough to catch sharp transients before they exceed the ceiling — a 1 ms attack means the limiter engages within approximately 48 samples at 48 kHz.
Release150 ms Matched to the compressor release so the two stages breathe at the same rate. A slower release (e.g. 500 ms) would reduce the average level noticeably; a faster release (e.g. 10 ms) would cause audible pumping on loud passages.

The limiter computes a per-sample target gain from the multi-channel peak, then smooths it with the attack/release coefficients. If the gain is falling (a new peak), the attack coefficient applies; if recovering, the release coefficient applies. This asymmetry means the limiter clamps hard and fast but lets go slowly — the correct behaviour for a peak limiter.

The pass also applies the mono mixdown at this stage when enabled. Rather than discarding one channel, both channels are averaged using vDSP_vadd + vDSP_vsdiv. Averaging preserves the level of signals that are identical in both channels (i.e. centre-panned speech) while reducing the level of signals that are out of phase between channels.

Pass 2.5
Silence shortening optional · −40 dBFS threshold · frame-accurate removal

Long pauses — a preacher pausing for effect, a gap while pages turn, silence before the benediction — are detected and trimmed to a configurable maximum duration (default 1 second). This runs after normalisation so the silence threshold is consistent regardless of the input recording level.

SettingValueWhy
Threshold−40 dBFS Matched to the slow leveler's noise floor gate. After voice isolation, residual room noise sits at roughly −40 to −45 dBFS, so −40 dBFS cleanly separates intentional silence from the noise floor without accidentally triggering on quiet breath or room tone.
Min silence duration0.5 s Pauses shorter than 500 ms are left untouched. Natural speech rhythm includes pauses of 200–400 ms between sentences; removing these would make the delivery sound unnaturally rushed. Only pauses that a listener would notice as "dead air" are candidates for shortening.
Max kept duration1.0 s (default) The user-configurable maximum. 1 second preserves enough pause to feel natural between sections (Bible reading → sermon, sermon → prayer) while removing the longer gaps that add nothing to the listener experience.

Removal is frame-accurate: the processor pre-computes which frame ranges to skip, then streams through the file writing only the frames outside those ranges. When a 4096-frame chunk straddles a boundary, sub-ranges within the chunk are written separately using memcpy into a scratch buffer. No resampling or cross-fading is applied — the edit is a hard cut, which is inaudible because the signal on both sides of the boundary is silence.

Chapter timestamps are remapped after shortening: each timestamp is adjusted by the total duration removed from silence segments that appear before it in the timeline. A chapter that falls inside a removed silent region snaps to the end of the kept tail.

Pass 3
MP3 encoding optional · bundled LAME · CBR

The final pass encodes the processed WAV to MP3 using a bundled, re-signed LAME binary. LAME is invoked as a subprocess with the --cbr flag (constant bitrate) rather than VBR, because podcast hosting platforms and some podcast apps handle CBR more reliably — particularly for chapter marker seek accuracy. The default bitrate is 128 kbps.

BitrateFile size (60 min)Use case
64 kbps~29 MBSpeech-only, bandwidth-constrained listeners. Audible compression artefacts on sibilants at this rate.
128 kbps~58 MBDefault. Clean speech quality. Transparent for voice at typical listening volumes.
192 kbps~86 MBHigher quality for services that re-encode (some podcast platforms transcode uploads).
256 kbps~115 MBNear-transparent. Overkill for speech but useful if the recording includes live music.

The LAME process output pipe is fully drained before waitUntilExit is called. This is necessary to prevent a deadlock: if LAME writes enough output to fill the pipe buffer (~64 KB on macOS), it blocks waiting for the pipe to be read; but waitUntilExit blocks waiting for the process to exit — neither side can proceed. Draining the pipe first ensures LAME can always write freely.

LAME is bundled rather than requiring a separate install because asking volunteer AV team members to install command-line tools is a meaningful adoption barrier. The binary is re-signed at build time with Hardened Runtime to satisfy macOS notarisation requirements.

YouTube downloads

yt-dlp management

DeChaff downloads audio from YouTube using yt-dlp — an open-source command-line tool that handles YouTube's ever-changing download API. It installs and updates yt-dlp automatically so volunteers never need to touch the terminal.

Why yt-dlp updates on every launch

yt-dlp's job is to work around YouTube's delivery API, which changes frequently and without notice. When YouTube rolls out a change, older versions of yt-dlp break and downloads fail with cryptic errors — the yt-dlp project typically ships a fix within hours. Checking for a new release on every launch means DeChaff is almost always running the latest version, and a volunteer whose download broke yesterday will find it working again the next time they open the app.

The check is fast: DeChaff fetches the latest release metadata from the GitHub API (api.github.com/repos/yt-dlp/yt-dlp/releases/latest, 10-second timeout) and compares the release tag against the version stored in UserDefaults. If the tags match and the binary passes a quick sanity check (yt-dlp --version exits 0), nothing is downloaded and the app proceeds immediately. An actual download only happens when the version has changed or the binary is missing or broken.

Where yt-dlp lives

yt-dlp (and ffmpeg, see below) are stored in the app's Application Support/DeChaff/ directory — outside the app bundle so they can be updated independently and survive app reinstalls. The binary is set to executable permissions (0o755) after every write, because macOS sometimes strips them when files are moved between directories.

Recent yt-dlp releases ship as a zip containing a PyInstaller onedir bundle — a top-level binary alongside a _Internal/ folder of Python modules. DeChaff unzips the whole bundle into Application Support, preserving _Internal/, and creates a canonical yt-dlp symlink if the zip named the binary yt-dlp_macos.

ffmpeg and live-stream VODs

YouTube live-stream recordings are stored in DASH format — split across many small fragments rather than a single file. yt-dlp needs ffmpeg to stitch these fragments together into a complete audio file. DeChaff downloads ffmpeg automatically the first time it's needed, sourcing a static build from the yt-dlp/FFmpeg-Builds GitHub repository. The correct architecture is selected at compile time (macos-arm64 or macos64) so the binary runs natively on both Apple Silicon and Intel.

If a system ffmpeg is already present (e.g. installed via Homebrew at /opt/homebrew/bin/ffmpeg) DeChaff falls back to it rather than downloading a second copy. The bundled version in Application Support always takes precedence.

Running yt-dlp: the PTY workaround

yt-dlp is a Python app bundled with PyInstaller. When run with its stdout connected to a Unix pipe (as is normal when spawning a child process from Swift), PyInstaller intercepts the output and silently swallows all stderr — including error messages and progress lines. This makes it impossible to show download progress or report errors to the user.

The fix is to run yt-dlp inside a pseudo-terminal (PTY) using /usr/bin/script. A PTY makes Python think it's writing to a real terminal, bypassing PyInstaller's pipe handling. script -q /dev/null yt-dlp … discards the transcript file (/dev/null) and passes PTY output through to its own stdout, which DeChaff reads via a pipe. ANSI escape sequences (cursor movement, colour codes) that a PTY emits are stripped with a regex before the lines are shown in the UI. 

// Wrap yt-dlp in /usr/bin/script to allocate a PTY — bypasses PyInstaller's
// pipe handling that swallows all stderr output when stdout is a pipe.
process.executableURL = URL(fileURLWithPath: "/usr/bin/script")
process.arguments = [
    "-q", "/dev/null",   // quiet mode; discard transcript file
    bin.path,              // yt-dlp binary
    "-x",                  // extract audio only
    "-f", "bestaudio[ext=m4a]/bestaudio/best",
    "--progress", "--newline",  // one progress line per update
    "-o", outputTemplate,
    videoURL
]

Progress lines ([download] 47.3% of …) are parsed with a regex to drive the progress bar.

Channel browsing: --flat-playlist

When a user enters a channel URL, DeChaff runs yt-dlp with --flat-playlist --dump-json to fetch metadata for the most recent videos without downloading any audio. Each video is returned as a JSON object on its own line, which DeChaff reads line-by-line from the pipe as they arrive (using PYTHONUNBUFFERED=1 to prevent Python's block-buffering from delaying output). Thumbnails are constructed directly from the video ID (img.youtube.com/vi/{id}/mqdefault.jpg) rather than using the often-absent thumbnail field in the JSON — YouTube's thumbnail URLs are predictable and reliable.

Transcription

On-device speech-to-text

After processing completes, DeChaff can generate a full transcript of the sermon using Apple's on-device speech recognition — privately, with no audio sent anywhere.

How it works

Transcription uses SpeechTranscriber and SpeechAnalyzer from Apple's Speech framework, introduced in macOS 26. These APIs replaced the older SFSpeechRecognizer and run entirely on-device using Apple Silicon's Neural Engine. Requires an Apple Silicon Mac running macOS 26+; the feature is hidden on unsupported hardware.

Before transcribing, DeChaff extracts only the trimmed region of the audio — the same window defined by the trim handles on the waveform — to a temporary CAF file. This avoids transcribing dead air or content that was cut. The temp file is deleted once the transcription task finishes.

The speech model may need to be downloaded the first time — macOS handles this automatically and it only happens once. DeChaff shows a status message while the download is in progress.

Progressive results

The SpeechTranscriber emits an async stream of results as it works through the audio. Each result has an isFinal flag: false results are tentative (the model is still deciding), while true results are committed and won't change. DeChaff accumulates finalized text and appends the latest tentative chunk, so the transcript appears to grow word-by-word in the UI as recognition progresses. 

var finalized = ""
var volatile = ""

for try await result in transcriber.results {
    try Task.checkCancellation()
    let chunk = String(result.text.characters)
    if result.isFinal {
        finalized += (finalized.isEmpty ? "" : " ") + chunk
        volatile = ""
    } else {
        volatile = chunk
    }
    // Show committed text + current tentative chunk
    let display = finalized + (volatile.isEmpty ? "" : " " + volatile)
    await MainActor.run { self?.transcriptText = display }
}

The results loop runs on a detached .utility priority Task so it doesn't block the UI.

What transcription is used for

The completed transcript is displayed in the Output step alongside the processed audio, where it can be copied to the clipboard. If the AI Assistant is enabled, the transcript is automatically sent to Claude as soon as transcription finishes — no manual step required.

AI Assistant

Automated post-production suggestions

Once a transcript is available, DeChaff can send it to Claude and ask for podcast titles, episode descriptions, and key themes — saving the AV team the copywriting step.

How it works

The AI Assistant is optional and off by default. When enabled in Settings → AI Assistant, it triggers automatically the moment transcription finishes. It sends the transcript to the Claude API using the same ClaudeAPIClient used by the metadata extraction Tier 2, with a customisable system prompt that also injects the sermon's metadata fields.

Requires a Claude API key stored in Settings → AI Assistant → Claude API Key (saved securely in the macOS Keychain). The Claude model used is configurable — any model ID can be entered; the default is Claude Sonnet 4.6.

The system prompt

The system prompt is fully editable in Settings. Before the prompt is sent, DeChaff substitutes four placeholders with the sermon's actual metadata: {title}, {preacher}, {series}, and {reading}. This gives the model context about the sermon without needing to infer it from the transcript alone.

The default prompt asks for:

  • A short podcast episode title (distinct from the sermon title)
  • A 2–3 sentence podcast episode description
  • 3–5 key themes or takeaways
You are a helpful church media assistant. Given the following sermon transcript, please suggest:
1. A short podcast episode title (not the same as the sermon title)
2. A 2-3 sentence podcast episode description
3. 3-5 key themes or takeaways

Sermon: "{title}" by {preacher}
Series: {series}
Bible Reading: {reading}

Placeholders are replaced with live values before the prompt is sent. The transcript is passed as the user message.

Customising the prompt

The prompt can be changed to anything — some teams use it to draft social media captions, generate sermon notes, or produce a summary for the church website. Because the transcript and metadata are always available as context, the prompt is the only thing that needs to change. A "Reset to Default" button in Settings restores the original prompt if needed.
