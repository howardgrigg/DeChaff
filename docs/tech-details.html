How DeChaff works

Implementation notes on the interesting parts — for contributors, curious minds, or churches adapting DeChaff for their own use.

Metadata extraction tiers Template parser
Metadata extraction

Three-tier YouTube title parsing

When a video is selected for download, DeChaff immediately tries to extract structured sermon metadata from the YouTube video title — sermon title, Bible reading, preacher, series, and date. It attempts three strategies in order, falling back to the next if the previous one is unavailable or fails. The extraction runs concurrently with the audio download, so the fields are pre-filled by the time the download finishes.

1
Apple Intelligence On-device · FoundationModels framework · macOS 26+
Preferred

The first attempt uses Apple's FoundationModels framework to run a small language model entirely on-device. DeChaff creates a LanguageModelSession with a system prompt that describes the expected title format (injected from the user's Settings → Templates configuration), then asks the model to populate a structured SermonMetadata schema using the @Generable / @Guide macros. The model returns a strongly-typed Swift struct directly — no JSON parsing required.

Because it runs on the Neural Engine, it adds no network latency and costs nothing per request. It is skipped if Apple Intelligence is unavailable on the device (checked via SystemLanguageModel.default.availability). 

// Structured output schema — the model fills this in directly
@Generable
struct SermonMetadata {
    @Guide(description: "The sermon title or topic.")
    var title: String
    @Guide(description: "Bible reading, e.g. 'Romans 8:28'.")
    var bibleReading: String
    @Guide(description: "Preacher or speaker name.")
    var speaker: String
    @Guide(description: "Sermon series name.")
    var series: String
    @Guide(description: "Date as 'dd MMM yyyy'.")
    var date: String
}

let session = LanguageModelSession(
    model: SystemLanguageModel.default,
    instructions: systemInstructions(titleFormat: titleFormat)
)
let response = try await session.respond(
    to: "Extract sermon metadata from: \"\(title)\"",
    generating: SermonMetadata.self
)
2
Claude API Network · Anthropic API · requires API key in Settings
Fallback

If Apple Intelligence is unavailable, DeChaff falls back to the Anthropic Claude API — the same API used by the AI Assistant feature. It sends the same system prompt (with the user's title format injected) and asks for a JSON response with the five metadata fields. The response is parsed from JSON into a SermonMetadata struct.

This tier requires an API key to be configured in Settings → AI Assistant and an active internet connection. It is skipped silently if no key is present. The model used is whatever is selected in Settings (defaulting to Claude Sonnet 4.6). 

// Same system prompt as tier 1, with a JSON output instruction appended
let response = try await ClaudeAPIClient.sendMessage(
    apiKey: apiKey,
    model: model,
    systemPrompt: systemInstructions(titleFormat: titleFormat) + """

        Respond with ONLY a JSON object:
        {"title":"...","bibleReading":"...","speaker":"...","series":"...","date":"..."}
        """,
    transcript: "Extract sermon metadata from: \"\(title)\""
)
3
Template parser On-device · pure string matching · no AI required
Last resort

If both AI tiers fail or are unavailable, DeChaff falls back to a deterministic regex parser that uses the user's title format template directly. Unlike AI, it cannot handle variation or imprecision — but when titles are consistently formatted it works reliably with zero latency and zero cost.

See Template parser below for a full explanation of how it works.

The shared system prompt

Tiers 1 and 2 use identical instructions. The user's title format is injected at runtime so the model understands the specific structure your church uses. This is why configuring the format in Settings → Templates matters — it significantly improves extraction accuracy for both AI tiers. 

"""
You extract structured metadata from church sermon YouTube video titles.

The expected format for this church is: "\(titleFormat)"
where {title} is the sermon title or topic, {reading} is the Bible reading or passage,
{preacher} is the speaker or preacher name, {series} is the sermon series name,
and {date} is the date of the sermon.

Fields may be separated by pipe characters (|), commas, dashes, or other delimiters
as shown in the format above. Not all fields are always present.

The date may appear in various formats (e.g. "23 Mar 2025", "15/3/26").
Always normalise it to "dd MMM yyyy" format.

Only populate fields you are confident about. Use empty string for anything not clearly
present. Never invent information.
"""
Tier 3

Template parser

The template parser converts the user's format string into a regular expression at runtime, then matches it against the YouTube title. Any separator the user configures works — the parser makes no assumptions about | or any other delimiter.

Walking through the default template as a concrete example:

Input template
{title}, {reading} | {preacher} | {series} | {date}
1

Find the placeholders

The regex \{(\w+)\} scans the format string and returns each {…} match in document order, capturing the field name inside the braces. 

match 0  {title}     name="title"
match 1  {reading}   name="reading"
match 2  {preacher}  name="preacher"
match 3  {series}    name="series"
match 4  {date}      name="date"
2

Build segments

For each placeholder match, the code slices out the literal text that sits between the previous match's end and this match's start. That literal is whatever separator the user typed — it could be , , | , - , — , or anything else. 

segment 0  literal=""     field="title"    ← nothing before {title}
segment 1  literal=", "   field="reading"  ← text between {title} and {reading}
segment 2  literal=" | "  field="preacher" ← text between {reading} and {preacher}
segment 3  literal=" | "  field="series"
segment 4  literal=" | "  field="date"
3

Build the regex

Each segment contributes two things to the pattern: its literal text regex-escaped (so | becomes \|, . becomes \., etc.), then a capture group. All capture groups are non-greedy (.+?) except the last, which is greedy (.+) so it consumes everything to the end of the string. 

segment 0  ""     →            (.+?)
segment 1  ", "   → ,\       + (.+?)
segment 2  " | "  → \ \|\    + (.+?)
segment 3  " | "  → \ \|\    + (.+?)
segment 4  " | "  → \ \|\    + (.+)   ← greedy on the last field

→ final pattern:
^(.+?), (.+?) \| (.+?) \| (.+?) \| (.+)$

Non-greedy groups are critical here — each (.+?) matches as little as possible, stopping at the first occurrence of the next literal rather than consuming it.

4

Progressive fallback for missing fields

If the title has fewer fields than the template — for example, no date — the full regex won't match because the expected separator and capture group aren't there. The parser retries with one fewer trailing segment, rebuilding the regex each time, until it either finds a match or runs out of options. A minimum of two matching fields is required to prevent the degenerate case of ^(.+)$ matching any arbitrary string. 

-- title: "Grace Abounding, Romans 5:1-5 | Pastor Mike | Core Series"
   (has series but no date)

count=5  ^(.+?), (.+?) \| (.+?) \| (.+?) \| (.+)$  → no match
count=4  ^(.+?), (.+?) \| (.+?) \| (.+)$           → match ✓

title    = "Grace Abounding"
reading  = "Romans 5:1-5"
preacher = "Pastor Mike"
series   = "Core Series"   ← captured by the greedy group
date     = ""               ← not in title, defaults to empty
5

Map captures to fields

Each capture group maps back to a field name via the used segments array, which preserves the original order. Each value is whitespace-trimmed, then assembled into a SermonMetadata struct.

Result — default template, all fields present
title→"The Good Shepherd"
reading→"John 10:1–18"
preacher→"Rev. James Hart"
series→"Foundations"
date→"6 Apr 2026"

Known limitations

Because the parser is purely structural, it has two inherent limitations that AI tiers do not:

Separator collision. If the separator character appears inside a field value — for example a sermon titled "Faith, Hope, and Love" when the separator between {title} and {reading} is , — the non-greedy capture will stop at the first comma and the mapping will be wrong. Tiers 1 and 2 handle this correctly because they understand context rather than pattern-matching literally.

Position-dependence. Fields must appear in exactly the order the template specifies. A missing field in the middle of the template (rather than at the end) will cause all subsequent fields to shift and map incorrectly. Missing trailing fields are handled gracefully by the progressive fallback.
